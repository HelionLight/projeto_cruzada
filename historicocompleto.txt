Análise do projeto (como funciona hoje)

Visão geral
O repositório projeto_cruzada é uma aplicação web feita com:

Backend: Node.js + Express
Banco: MongoDB + Mongoose
Arquivos: upload e armazenamento de documentos/fotos via GridFS
Autenticação admin: JWT
Frontend: arquivos estáticos em public/ (HTML/CSS/JS)
O fluxo central do sistema é: cadastro → temporário (pendente) → revisão/aprovação → move para banco permanente (aprovado) → gestão/edição/exportação.

Estrutura do código
1) server.js (servidor principal)
Responsável por:

ler variáveis do .env (dotenv)
configurar middleware do Express (cors, express.json, urlencoded)
conectar no MongoDB via mongoose.connect(MONGO_URI)
registrar as rotas:
app.use('/api/auth', require('./routes/auth'))
app.use('/api/cruzados', require('./routes/cruzados'))
servir o frontend:
express.static(.../public)
rota / serve public/index.html
rota /admin serve public/admin.html
ter um middleware de erro global (retorna 500)
2) Rotas de autenticação: routes/auth.js
Endpoint principal:

POST /api/auth/login
recebe username e password
busca User no MongoDB
valida senha via user.comparePassword
gera JWT com process.env.JWT_SECRET
retorna { token }
3) Middleware de segurança: middleware/auth.js
Tem dois papéis:

authenticate

lê Authorization: Bearer <token>
faz jwt.verify
carrega req.user = User.findById(decoded.id)
bloqueia com 401 se token inválido
authorize(...roles)

checa req.user.role estar entre os roles aceitos
bloqueia com 403 se não tiver permissão
4) Modelos (MongoDB schemas)
models/User.js
username (unique)
password (hash)
role (admin/secretario/tesoureiro)
middleware pre('save') para hash com bcryptjs
método comparePassword
models/Cruzado.js
Schema do cadastro permanente:

dados pessoais e profissionais
foto, certificadoIndicacao, documentoConsignacao, documentoVoluntario como ObjectId (referências aos arquivos no GridFS)
status: pendente | aguardando_documentos | aprovado | rejeitado
Observação importante: a lógica de temporário/aprovado usa também um modelo “temp”.

5) Modelo temporário: models/CruzadoTemp.js
Eu não li o arquivo ainda, mas pelo uso nas rotas ele é a coleção temporária que recebe cadastros novos com status pendente e depois avança para revisão.

6) Rotas do cadastro/admin: routes/cruzados.js
Esse arquivo concentra praticamente toda a lógica de negócio.

6.1 Configuração de upload (GridFS + multer)
usa multer.memoryStorage() (fica em memória)
limita tamanho do arquivo: 5MB
define campos:
foto
certificadoIndicacao
documentoVoluntario
documentoConsignacao
Quando chega um cadastro, ele salva cada arquivo no GridFS:

getGridFSBucket()
bucket.openUploadStream(nomeOriginal, { contentType })
uploadStream.end(buffer)
guarda o uploadStream.id no Mongo no campo do schema (foto, etc).
6.2 Envio de e-mail para o secretário (já existe)
Há uma função enviarAlertaSecretario() que:

habilita/desabilita via EMAIL_ALERT_ENABLED
cria transportador SMTP via variáveis:
SMTP_HOST, SMTP_USER, SMTP_PASS, (opcional SMTP_PORT, SMTP_SECURE)
conta pendentes em CruzadoTemp onde status está em ['pendente','aguardando_documentos']
envia e-mail para:
SECRETARIO_EMAIL ou EMAIL_USER ou fallback 'secretaria@cme.org.br'
E isso acontece dentro da rota de register (após salvar).

6.3 Validação do formulário (Joi)
cruzadoSchema define campos obrigatórios e enums.
Ex.: sexo tem que ser masculino|feminino, vinculoProfissional tem lista fechada, etc.

Além disso, há um “sanitizer” que:

remove campos vazios ''
converte strings "true"/"false" em boolean
converte valorContribuicao em número
6.4 POST /api/cruzados/register (cadastro novo)
Fluxo:

valida schema Joi
checa duplicidade em CruzadoTemp:
CPF duplicado → 400
Email duplicado → 400
salva arquivos enviados no GridFS (se existirem)
cria new CruzadoTemp(cruzadoData) e salva
chama enviarAlertaSecretario()
responde 201 Registro enviado para aprovação
6.5 GET /api/cruzados/pending (admin/secretário)
retorna CruzadoTemp com status: 'pendente', ordenado por createdAt
6.6 GET /api/cruzados/pending/voluntarios
retorna temporários com:
status: 'aguardando_documentos'
trabalharVoluntario: true
e documentoVoluntario existente
6.7 PUT /api/cruzados/:id/status (aprovar/rejeitar)
Recebe { status: 'aprovado' | 'rejeitado' }

Se aprovado:
verifica se precisa de revisão documental:
trabalharVoluntario || documentoConsignacao || documentoVoluntario
se precisa:
atualiza temporário para aguardando_documentos
caso contrário:
cria new Cruzado({...temp, status:'aprovado'})
salva no banco permanente
deleta o registro temp (CruzadoTemp.findByIdAndDelete)
Se rejeitado:
marca no temp status:'rejeitado' (não remove na hora)
comentário indica “será deletado pelo TTL” (TTL precisa existir na coleção/índices)
6.8 Atualização/edição e exclusão (admin)
PUT /api/cruzados/:numeroCruzado:
findOneAndUpdate({numeroCruzado})
DELETE /api/cruzados/:numeroCruzado:
deleta permanente por numeroCruzado
6.9 router.put('/atualizar/:id') (edição pelo usuário)
Essa rota:

valida dados (Joi)
busca por id (ObjectId)
evita duplicidade de CPF/email comparando “CPF sem formatação”
processa upload de foto, certificadoIndicacao, documentoConsignacao
atualiza Cruzado com os novos IDs de GridFS
6.10 Arquivos: GET /api/cruzados/image/:id
recebe um id de GridFS (24 chars)
abre stream download do GridFS e faz pipe(res)
6.11 Consignação e exportação
GET /api/cruzados/consignacao: lista temporários com consignação + documento
GET /api/cruzados/export/excel: exporta Cruzado.find({status:'aprovado'}) para XLSX
6.12 Buscar por CPF
GET /api/cruzados/buscar?cpf=...
tenta busca direta (cpf: cpfQuery)
depois busca por versão sem não-dígitos
fallback usa $regexReplace (se suportado no Mongo da stack)
Como o sistema “anda” na prática (fluxo completo)
Usuário preenche formulário e envia arquivos → POST /register
Sistema cria registro em CruzadoTemp (pendente) e salva os arquivos no GridFS
Envia e-mail para a secretaria com quantidade de pendentes

Secretário aprova:
pode mandar para aguardando_documentos ou diretamente mover para Cruzado com status aprovado
Se aprovado e não precisar documentos → o temp é apagado
Exportação e edição usam a coleção permanente Cruzado
